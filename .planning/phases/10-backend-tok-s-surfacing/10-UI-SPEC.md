# Phase 10 ‚Äî UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Scope of This Contract ‚Äî read first

This is **NOT a greenfield UI**. The VillaStraylight control dashboard already exists and is **locked to a vanilla-JS, no-framework architecture** (Phase-5 D-01). The entire UI surface is exactly three embedded files, edited in place and served via `internal/dashboard/embed.go` (`go:embed all:assets`):

- `internal/dashboard/assets/dashboard.html`
- `internal/dashboard/assets/dashboard.css`
- `internal/dashboard/assets/dashboard.js`

**Hard constraints (do not violate):** NO React / NO build step / NO new dependencies / NO new HTTP endpoints / NO external fonts, CDNs, analytics, or any network call beyond the existing same-origin `/api/*` polls (strictly-local / zero-telemetry, project-wide).

The dashboard polls `/api/status` (`dashboard.js:447`) and `/api/metrics` (`dashboard.js:464`); `renderPerformance` (`dashboard.js:149`) fills the Performance panel; `renderHealth` (`dashboard.js:110`) fills the Health panel.

**This contract therefore specifies ONLY the three append-only additions for Phase 10**, each reusing the existing panel / metric-row / badge idioms already in the codebase. The design tokens below are **transcribed from the existing `dashboard.css`** ‚Äî they are not new inventions; they are the contract the additions must obey.

### The three NEW visual elements (all append-only, all honest-Unknown)

| # | Element | Panel | Data source | Honesty rule |
|---|---------|-------|-------------|--------------|
| 1 | **Active backend + image tag** | Health | `/api/status` poll: `report.backend` + `report.image` (mirrors `villa backend show` `{backend, image}`) | Image unset ‚Üí omit the image row; backend unset ‚Üí gray "unavailable" badge, never a literal default |
| 2 | **Live tok/s labeled by backend** | Performance | tok/s number from `/api/metrics` (unchanged); backend label from the `/api/status` poll (D-01 ‚Äî identity lives in status, NOT metrics) | Idle/unavailable keep the EXISTING honest copy ("Idle ‚Äî no active generation." / "Unavailable"); never append a backend to a fabricated `0` |
| 3 | **ROCm-readiness indicator** | Health | `/api/status` poll: `report.rocm_readiness` tri-state (`ready` / `not-ready` / `unknown`) | Unevaluable / off-hardware ‚Üí gray `unknown` badge (reuse the GPU panel's "Unavailable"-badge precedent); NEVER a red `not-ready` from an unevaluable signal |

### Project-wide honesty constraint (CRITICAL ‚Äî checker must enforce)

The ROCm advantage is **prompt-processing-weighted**; **token-generation is roughly flat and may regress** (on-hardware UAT: Œîpp +4.84 / Œîtg ‚àí11.15). **No UI copy may imply ROCm is "faster."** Readiness ‚âÝ a speed promise. The `recommend` ROCm advice is a **CLI surface (not the dashboard)**; if any dashboard copy references it, it must point to "verify with `villa bench`" and must NEVER promise a speed-up.

## Typography

Transcribed from `dashboard.css` (font-size declarations at `:45`, `:77`, `:114`, `:140`, `:159`). The dashboard uses **4 roles, 2 weights** ‚Äî Phase-10 additions must reuse these exactly; introduce no new size or weight.

| Role | Size | Weight | Line Height | CSS reference |
|------|------|--------|-------------|---------------|
| Display | 28px | 600 (semibold) | 1.2 | `.header-title` (`:77`) ‚Äî not used by these additions |
| Heading | 20px | 600 (semibold) | 1.2 | `.panel-heading` (`:114`) ‚Äî existing Health/Performance headings, unchanged |
| Body | 16px | 400 (regular) | 1.5 | `html, body` base (`:45`) ‚Äî backend name, row labels |

## Copywriting Contract

All copy below is the exact string to render. Server values (backend name, image tag, tok/s number) are interpolated via `textContent` only (XSS-safe; never `innerHTML`).

| Element | Copy |
|---------|------|
| Primary CTA | **None** ‚Äî Phase 10 adds zero interactive controls (all three additions are read-only display). The dashboard's only CTA, "Switch model", is out of scope and unchanged. |
| Active backend row (element 1) | Label: `backend` ¬∑ Value: the resolved name verbatim, e.g. `vulkan` or `rocm` (from `report.backend`). |
| Active image row (element 1) | Label: `image` ¬∑ Value: the resolved image tag verbatim (from `report.image`), monospace tabular. |
| Backend unknown state (element 1) | If `report.backend` is absent/empty: a gray `badge badge-unknown` reading `unavailable` (reuse `healthLabel` default). Never render a fabricated default backend name. |
| tok/s ‚Äî generating (element 2) | `generation` ‚Üí `{N.N} tok/s ({backend})`, e.g. `12.3 tok/s (vulkan)`. Number from `/api/metrics` `gen_tokens_per_sec`; `({backend})` suffix appended ONLY when `lastBackend` is known. |
| tok/s ‚Äî idle (element 2) | Keep existing: `Idle ‚Äî no active generation.` ‚Äî **do NOT append a backend.** |
| tok/s ‚Äî activity unknown (element 2) | Keep existing: `Activity unknown ‚Äî slot status unavailable.` |
| tok/s ‚Äî unavailable (element 2) | Keep existing: `Unavailable` |
| ROCm-readiness badge (element 3) | `ROCm ready` (green) / `ROCm not ready` (amber) / `ROCm readiness unknown` (gray) per the mapping table above. |
| Empty / unknown state (element 3) | The `unknown` badge IS the honest empty state; optional caption via `mutedP`: `ROCm readiness can't be evaluated on this host.` (mirrors the GPU panel's "GPU utilization isn't reliably reported on this hardware." idiom). |
| Error state | Per-panel failure already handled by the existing poll loop (D-11): Health drives the global red connection banner ("Lost connection to the dashboard. Retrying‚Ä¶"); Performance degrades independently to `Unavailable`. **Phase 10 adds no new error path** ‚Äî the new rows simply omit when their source field is absent (stale-while-revalidating dims them via `.stale`). |
| Destructive confirmation | **None** ‚Äî Phase 10 has no destructive actions. The only write (`/api/models/switch`) is untouched. |

**Forbidden copy (checker must fail on any of these in the dashboard assets):** any string implying ROCm is "faster", a "speed-up", "better performance", "recommended over vulkan", or "guaranteed". Readiness copy states readiness only. (The honest "verify with `villa bench`" advice is a CLI `recommend` surface, not the dashboard.)

**Seam-gate copy rule (CI ‚Äî `TestSeamGrepGate`, `seam_test.go:34`):** the bare word `ROCm`/`rocm` in a badge label is allowed; the gated device/marker literals are NOT ‚Äî never write `ROCm0`, `HSA_OVERRIDE_GFX_VERSION`, `Memory access fault`, `kyuz0`, `docker.io/`, `server-vulkan`, `rocm-7.2.4`, `--device`, `--group-add`, or `keep-groups` into any dashboard asset. The backend name and image tag arrive as runtime values from `report.backend`/`report.image` (returned by the inference seam), not as literals in the JS/HTML.

## Accessibility (existing conventions ‚Äî additions must honor)

- The overall verdict badge uses `aria-live="polite"` (`dashboard.html:19`); the readiness badge sits in the Health panel and updates on poll ‚Äî no new live-region needed, but its text must change with state (not color alone) ‚Äî the tri-state copy already encodes meaning textually.
- Status is **never color-only**: every badge carries text (`ready` / `not ready` / `readiness unknown`); the dot is supplementary. Maintain this for element 3.
- `@media (prefers-reduced-motion: reduce)` disables transitions globally (`dashboard.css:344`) ‚Äî additions add no animation.
