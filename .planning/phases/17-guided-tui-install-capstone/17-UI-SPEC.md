# Phase 17 — UI Design Contract (Terminal TUI)

> Visual and interaction contract for the guided `villa install` wizard. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **This is a TERMINAL TUI, not a web/browser frontend.** Built with `charmbracelet/huh` v1.0.0 (forms) + `lipgloss` v1.1.0 (styling) + `bubbletea` v1.3.6, rendered in a Unix terminal. There is NO HTML/CSS, NO browser, NO responsive breakpoints, NO mouse-first interaction, NO web fonts. Every dimension below is expressed in terminal-UI terms (ANSI/lipgloss color tokens, terminal rows/columns, bold/faint/underline, keyboard navigation). Where a web-oriented dimension does not map, it is explicitly translated to its TUI equivalent rather than dropped.
>
> The wizard is **pure presentation** (INSTALL-01, D-domain). It composes existing cores (`recommend.Pick`, `internal/preflight`, `inference.BackendFor`) and the existing `installDeps` seam; it computes no fit/preflight/backend decisions itself. It lives entirely in `cmd/villa/`; no `internal/*` core imports the TUI library (D-11).

## Design System

| Property | Value |
|----------|-------|
| Tool | none (terminal TUI — shadcn / web design systems do not apply) |

## Spacing Scale

**Web → TUI translation.** There are no px/rem in a terminal. The "spacing scale" is **terminal cells**: blank rows (vertical rhythm) and leading spaces / lipgloss padding (horizontal indent). All values are whole character cells. huh manages most intra-field spacing; first-party spacing tokens govern the wizard chrome (header, footer, notes, review block).

Declared values (terminal cells — the TUI analogue of a spacing scale):

| Token | Value | Usage |
|-------|-------|-------|
| tight | 0 rows | Between a label and its inline help/hint |
| step | 1 blank row | Between stacked fields within a screen; between a Note title and its body |
| block | 1 blank row + 2-cell left indent | Indented detail lines under a heading (e.g. each review line, each preflight gap) |
| screen | 1 blank row above + 1 below | Vertical padding around the persistent header and the key-hints footer |
| gutter | 2 cells | Standard left indent for body content beneath a screen heading |
| marker-gap | 1 cell | Between a status glyph (●/✓/!/✗) and its text |

Exceptions:
- huh's own internal field padding (selector caret, option list indent) is owned by huh and is NOT re-specified here — do not fight the library's layout.
- The "Step N/M" progress indicator sits on the header row with a 1-cell gap before the screen title; no extra blank row between them.

## Typography

**Web → TUI translation.** A terminal has no font sizes or weights — only attributes the terminal can render: **bold**, **faint/dim**, **underline**, and reverse/inverse. The "type scale" is therefore an **emphasis scale** built from lipgloss attributes. Hierarchy is conveyed by attribute + color + indentation, never by px. All four roles MUST remain legible and distinguishable under `NO_COLOR` / `TERM=dumb` using attributes alone (D-09).

| Role | Terminal attributes | Color token | Usage |
|------|--------------------|-------------|-------|
| Display (screen title) | **bold** + underline | accent (degrades to bold+underline mono) | The one title per wizard screen (e.g. "Confirm your model") |
| Heading (section/group) | **bold** | default fg | Sub-section labels within a screen (e.g. "Preflight results", "Will install") |
| Body / label | regular (no attribute) | default fg | Field prompts, option text, review values |
| Help / hint (footer + field help) | **faint/dim** | muted fg | Key-hints footer, inline field help, "Step N/M", secondary detail |

Rules:
- **At most these 4 emphasis roles.** Do not invent a 5th (no blink, no reverse-video for hierarchy — reverse is reserved by huh for the focused option cursor).
- Hierarchy must survive theme-off: under `NO_COLOR`/`TERM=dumb`, Display still reads as bold+underline, Heading as bold, Help as faint; if the terminal cannot render faint, Help degrades to plain text and hierarchy falls back to indentation + the glyph column.
- Status words (BLOCK/WARN/PASS) are **bold** in addition to their status color so they remain scannable when color is stripped.

## Color

**Web → TUI translation.** No hex/CSS variables drive runtime rendering — colors are **lipgloss color tokens** (`lipgloss.AdaptiveColor{Light, Dark}` so the same theme reads on light and dark terminals; ANSI 16/256 values for portability). The 60/30/10 split is reframed for a terminal: the terminal's own background is the dominant surface (villa does not paint a full-screen canvas), text default is the secondary, and villa's **accent + status palette is the deliberately scarce ~10%** reserved for the specific elements listed below. The hex values in the table are **theme-authoring references** for `lipgloss.AdaptiveColor`, not CSS.

| Role | Token (lipgloss) | Reference value | Usage |
|------|------------------|-----------------|-------|
| Dominant (60%) — surface | terminal default background (unpainted) | n/a (host terminal) | The wizard never sets a full-screen background; it renders on the user's terminal canvas |
| Secondary (30%) — body text | terminal default foreground | n/a (host terminal) | All body/label/value text inherits the terminal's default fg |
| Muted (help/faint) | `AdaptiveColor{Light:"244", Dark:"245"}` | ~#808080 grey | Faint help text, "Step N/M", inline field help, secondary detail |
| Accent (10%) | `AdaptiveColor{Light:"63", Dark:"105"}` indigo/violet | ~#7D56F4 | RESERVED — see list below |
| Status: PASS | `AdaptiveColor{Light:"34", Dark:"42"}` green | ~#2E8540 / #43BF6D | A preflight check that PASSed; the final "Installed" success line |
| Status: WARN | `AdaptiveColor{Light:"130", Dark:"214"}` amber | ~#B0730D / #FFAF00 | A WARN-tier preflight gap; skew/advisory notices |
| Status: BLOCK | `AdaptiveColor{Light:"160", Dark:"196"}` red | ~#D70000 / #FF0000 | A BLOCK-tier preflight gap; fatal/abort messages |

Accent reserved for (explicit list — NEVER "all interactive elements"):
1. The focused-field selector caret / cursor in huh Select & Confirm (`Focused.SelectSelector`, focused button).
2. The screen Display title (the one title per screen).
3. The "Step N/M" progress indicator's current-step emphasis.
4. The final-review screen's primary **Install** confirm button when focused.

Status colors (PASS=green / WARN=amber / BLOCK=red, D-10) are reserved EXCLUSIVELY for preflight/result status semantics and the terminal success/abort lines — never decorative. Status color always co-occurs with a bold status word AND a status glyph so meaning survives `NO_COLOR`.

**Mandatory degradation (D-09 — theme-only, never flow):**
- If `NO_COLOR` is set OR `TERM=dumb` OR stdout is detected non-color: build the theme from a **no-color variant** (all `Foreground`/color calls stripped to the zero `lipgloss.Style`), keeping bold/faint/underline attributes and the glyph column. The guided wizard STILL runs end-to-end — only styling is removed (D-09 is explicit: degrade the THEME, not the wizard).
- The accent and the three status colors are the ONLY runtime colors. No gradients, no background fills, no 24-bit-only colors that fail on 256-color terminals.

## Iconography & Glyphs (TUI-specific dimension)

**Web → TUI translation of "icon library".** No SVG/icon font. Status and structure are shown with a small fixed glyph set, each with a mandatory ASCII fallback for non-UTF-8 terminals (detect via locale / `TERM`; when in doubt, fall back to ASCII).

| Meaning | Unicode | ASCII fallback | Color |
|---------|---------|----------------|-------|
| PASS | `✓` | `[OK]` | PASS green |
| WARN | `!` (U+0021, or `▲`) | `[WARN]` | WARN amber |
| BLOCK | `✗` | `[BLOCK]` | BLOCK red |
| Selected / current option | `▸` (huh selector) | `>` | accent |
| Bullet / detail line | `•` | `-` | default fg |
| Progress filled / current step | `●` | `*` | accent |

Rule: glyph + bold status word together carry meaning; color is additive only. Never rely on color alone to distinguish PASS/WARN/BLOCK.

## Layout & Screen Sequence (TUI-specific dimension)

**Web → TUI translation of "page layout".** No DOM, no responsive grid. Layout is **terminal rows top-to-bottom**, one wizard screen at a time, with a persistent header and footer. Each screen is a huh `Group` (or a `Note` + a field). The wizard composes the named cores; it adds zero decision logic (INSTALL-01, SC#2).

**Persistent chrome (every screen):**
- **Header row:** `villa install` (faint) · `Step N/M` progress indicator (accent on current step) · screen Display title (accent, bold+underline). M is the fixed total of wizard steps.
- **Body:** the screen's field(s) / note, indented at the `gutter` (2 cells).
- **Footer (key-hints, faint):** the active key map for this screen (see Interaction).

**Screen sequence (5 steps — Claude's-discretion decomposition, honoring D-07's final confirm):**

| Step | Screen | huh widget | Core composed | Notes |
|------|--------|-----------|---------------|-------|
| 1/5 | Detected host | `huh.NewNote` (read-only summary) | `detect` (host probe) | Shows CPU, memory envelope, iGPU, kernel, backend=Vulkan (D-03). Typed-Unknown facts render faint as "unknown", never a fabricated 0. Continue to advance. |
| 2/5 | Confirm / adjust model | `huh.NewSelect` | `recommend.Pick` | Options = recommended pick (pre-highlighted, labelled "recommended") + other catalog picks that fit the detected envelope, ALL re-validated by `recommend.Pick` (D-02). Each option line: model · quant · ctx · fit. Wizard computes nothing. |
| 3/5 | Preflight gate | `huh.NewNote` for results + per-item `huh.NewConfirm` for privileged fixes | `internal/preflight` + `gateInstall`/`resolveGap`/`runGapFix` | See Preflight Gap States below. Auto-runs safe fixes with a visible notice (D-04/D-05); privileged fixes get per-item explicit consent. BLOCK unmet → cannot proceed without `--force`; 0/2/1 exit contract preserved (D-06). |
| 4/5 | Final review & confirm | `huh.NewConfirm` ("Install" / "Cancel") | composes prior selections | The single explicit gate before ANY host mutation (D-07). Lists chosen model/quant/ctx/backend + exactly what will be pulled / written / started. |
| 5/5 | Installing → result | progress notes (non-huh streaming) + terminal status line | existing `runInstall` host effects | Reuses the existing install execution + `printPostInstall`. Final line is a PASS-green success or a BLOCK-red abort. |

Layout rules:
- One screen owns one decision. No horizontal split panes (terminal-portable; avoids width assumptions).
- Content must render legibly at **80 columns** (the portability floor); wrap long lines, never assume a wide terminal. Do not hard-require a minimum height beyond the visible group.
- The header `Step N/M` indicator is mandatory and increments monotonically; M is constant for a given run.

## Interaction Contract (TUI-specific dimension)

**Web → TUI translation of "interaction states".** No hover, no click, no focus-ring CSS. Interaction is **keyboard-only**, driven by huh's built-in key handling. The contract below is the consistent key map surfaced in the footer.

| Key | Action | Scope |
|-----|--------|-------|
| `↑` / `↓` (and `k` / `j`) | Move selection between options | Select screens |
| `Tab` / `Shift+Tab` | Move between fields / focus Confirm buttons | within a group |
| `Enter` | Confirm the focused field / advance to next screen | all screens |
| `y` / `n` | Answer a Confirm (privileged-fix consent; final Install) | Confirm screens |
| `Esc` (or `Ctrl+C`) | Abort the wizard cleanly with NO host mutation; exit non-zero; print how to resume / use `--no-tui` | all screens (before step 5 mutation point) |

Focus & state model:
- Exactly one field is focused at a time (huh `Focused` vs `Blurred` styles); the focused selector caret is accent-colored.
- The **Install** confirm (step 4) defaults focus to a non-destructive choice (Cancel is the safe default) so an accidental Enter does not mutate the host — install requires a deliberate selection of "Install" then Enter (D-07, "never silently mutate").
- Validation errors (e.g. an out-of-envelope manual override path, if reached) render inline via huh's `ErrorIndicator`/`ErrorMessage` (BLOCK red + glyph), never as a crash.

**Graceful degradation / fallback (D-01/D-08 — flow-level, distinct from D-09 theme-level):**
- Launch the wizard ONLY when: stdin AND stdout are a TTY, AND `--no-tui` is NOT set, AND `--json` is NOT set. Otherwise run today's flag-driven `runInstall` path verbatim — flags stay first-class (D-08, SC#3).
- `--no-tui` is an explicit opt-out to the flag path; there is NO `--tui` opt-in and NO new `villa setup` subcommand — one progressively-enhanced command (D-01).
- `--json` always implies non-interactive → flag path (a `--json` run is never a wizard).
- The non-TTY/`--no-tui` path produces the SAME `config.toml` and runs the SAME install as the wizard (SC#1/SC#2) — the wizard is a front-end, not a different install.

## Preflight Gap States (TUI-specific state matrix)

The step-3 gate renders `internal/preflight` `CheckResult`s. Each gap row = glyph + bold status word (color) + check name + remediation text. State handling mirrors the existing `install.go` gap machinery — the wizard only presents it.

| State | Render | Behavior |
|-------|--------|----------|
| PASS | `✓ PASS` (green) + check name | Informational; advance. |
| WARN (no auto-fix) | `! WARN` (amber) + name + remediation | Surfaced as advisory; wizard proceeds (boot-time soft gap, never blocks). |
| WARN/BLOCK with **safe (non-privileged) auto-fix** | `! …` + "auto-fixing: <cmd>" notice (faint) | Auto-runs with a VISIBLE notice — no prompt (D-04/D-05). Applies to BOTH TUI and flag path. Respects non-interactive mode. |
| BLOCK with **privileged (sudo) fix** | `✗ BLOCK` (red) + the exact privileged command + a `huh.NewConfirm` per item | Requires EXPLICIT per-item `y` consent before running (D-04). villa NEVER silently runs a privileged command. Decline → gap stays unmet. |
| BLOCK unmet (declined / no fix) | `✗ BLOCK` (red) + remediation + "use --no-tui … --force to override" | Cannot proceed without `--force`; preserves the 0/2/1 (PASS/BLOCK/WARN) exit contract (D-06). Wizard aborts to that exit code, no mutation. |

## Copywriting Contract

Terminal copy: concise, lowercase-command-aware, remediation-forward. No marketing voice; imperative and specific.

| Element | Copy |
|---------|------|
| Primary CTA (final apply) | **Install** (the focused confirm on step 4/5; sibling choice: **Cancel**) |
| Step-2 confirm CTA | **Use this model** (advance from the Select) |
| "Empty state" — no fitting model | `No catalog model fits the detected memory envelope (<N> GiB usable). Free memory or supply a larger-envelope host, then re-run villa install. (--no-tui shows the same result.)` |
| "Empty state" — detect typed-Unknown | Render the unknown fact faint as `unknown` with a one-line note: `Some host facts could not be probed; villa will pick conservatively. Run villa detect for detail.` (never a fabricated 0) |
| Error state — BLOCK gap declined | `BLOCK: <check name>. <remediation>. Run the suggested command, or re-run with --no-tui --force to override (auditable).` |
| Error state — wizard abort (Esc/Ctrl+C) | `Install cancelled — no changes were made. Re-run villa install, or villa install --no-tui for the flag-driven path.` |
| Review screen heading | `Review — villa will install:` followed by indented lines: `model/quant/ctx`, `backend: vulkan`, `will pull: <images>`, `will write: <quadlet units>`, `will start: <services>`. |
| Destructive confirmation — Install (host mutation) | `Install: this will pull images, write Quadlet units, and start services. Proceed?` → focus defaults to **Cancel**; choosing **Install** + Enter is the single explicit gate (D-07). |
| Success line | `✓ Installed — <model> on vulkan. Chat: http://127.0.0.1:<chat_port>  Dashboard: http://127.0.0.1:<dashboard_port>` (reuses `printPostInstall`). |

Copy rules:
- Backend is always rendered as `vulkan` by default and is NEVER auto-switched in copy or behavior (D-03). The wizard never offers a silent backend flip.
- Backend/image NAMES are rendered via existing accessors behind the `internal/inference` seam — never re-typed literals in TUI code (`TestSeamGrepGate`).
- Every non-PASS state carries actionable remediation (project-wide refuse-with-remediation rule).

## Theme File Contract (executor reference)

One command-tier theme file (D-10) is the single source of TUI styling:
- Exposes a constructor returning `huh.Theme` (or `*huh.Styles`) built from `huh.ThemeBase(isDark)` with villa overrides: accent on `Focused.Title`/`Focused.SelectSelector`/focused button; the three status colors as named lipgloss styles reused for preflight rows and the result line; faint help.
- Takes a `colorEnabled bool` (computed from `NO_COLOR` / `TERM=dumb` / non-TTY-color detection) and, when false, returns a styles object with all `Foreground` calls stripped while RETAINING bold/faint/underline + glyph column (D-09).
- Exposes the `Step N/M` header renderer and the status-glyph helpers (Unicode + ASCII fallback) so every screen draws chrome consistently.
- Is testable off-hardware via the existing seam pattern (`installDeps`/`live*Deps`/`fake*Deps`): the wizard's TTY/color/consent inputs are injected funcs, so the whole flow runs in tests with no live terminal.
