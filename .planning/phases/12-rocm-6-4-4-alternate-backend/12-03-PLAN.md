<objective>
Close the one genuine design gap for SC#3: `bench --ab`'s `other(orig)` is a hardcoded 2-value
swap that cannot A/B two ROCm backends (e.g. `rocm-6.4.4` vs `rocm-7.2.4`). Implement Option A
(the RESEARCH-recommended choice, selected per CONTEXT --auto mode): add an optional explicit
A/B target so the user can prove which digest recovers the v1.1 Δtg −11.15 regression against
either rocm-7.2.4 OR Vulkan. Then run the on-hardware checkpoint (switch + residency proof +
bench) that off-hardware tests cannot cover (SC#1/SC#3 are partly host-only).

Purpose: SC#3 says "measures the new image against rocm-7.2.4 / Vulkan" — Option A directly
satisfies the rocm-7.2.4 leg (arbitrary-pair A/B), which the v1.1 binary `other()` cannot
express for a 3-ROCm world. `bench.Run` already takes the flip target via the injected `Switch`
closure, so this is a small, well-bounded change (one flag + one spec field + one core plumb).
Output: arbitrary-pair `bench --ab`; on-hardware proof of transactional switch + Δtg recovery.

**Alternative considered (Option B, NOT chosen):** ship the additive backends and document the
v1.1 binary model — the user proves Δtg by setting each ROCm backend active and `--ab`-ing vs
Vulkan, then comparing the two Δtg figures. Rejected because it satisfies the rocm-7.2.4 leg of
SC#3 only indirectly (two separate vs-Vulkan runs); --auto mode selects the recommended Option A.
</objective>
