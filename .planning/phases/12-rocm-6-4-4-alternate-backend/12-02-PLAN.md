<objective>
Generalize the ROCm-family routing and make the policy gate meaningful for the new images:
replace every literal `"rocm"` comparison with `inference.IsROCmFamily` (D-08), thread the
resolved target image into the preflight policy gate so it evaluates the ACTUAL 6.4.4 digest
against `imageDeny` (Pitfall 3 / SC#2), widen `orchestrate.backendLabel` so a 6.4.4 unit is
not mislabeled "Vulkan RADV" (Pitfall 2), and widen the detect `rocmImagePolicyOK` so a 6.4.4
host reports honest `rocm_readiness.image_policy_ok` (Pitfall 3).

Purpose: SC#2 ("gated by rocm-policy.json floors; a request that fails a floor is refused with
named remediation, never silently downgraded") is only true for the new backends once the gate
fires for them AND evaluates their real image. The literal `== "rocm"` checks would silently
skip the gate for the new names. This plan closes every call-site from the RESEARCH audit.
Output: All ROCm-family backends gated identically; honest labels + readiness; off-hardware green.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| hand-edited config.toml → PreflightROCm | The `backend` value decides whether the ROCm refuse-with-remediation gate runs; a family-blind check would silently skip the gate for a new ROCm name. |
| resolved target image → policy gate | The gate must evaluate the ACTUAL image against imageDeny, not an empty string, or a denied image could pass unevaluated. |
| detect seam → status/JSON | The `image_policy_ok` readiness literal is allowed only in the detect seam file (gpu_amd.go). |

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-12-05 | Tampering | denied image passing the gate unevaluated | mitigate | `RunROCmForImage` threads the resolved digest so `checkROCmImage` evaluates it against `imageDeny` (Pitfall 3 / SC#2) — no empty-image WARN bypass. |
| T-12-06 | Repudiation | misleading "Vulkan RADV" Description on a ROCm unit / "unknown image" readiness | mitigate | Widen `backendLabel` + `rocmImagePolicyOK` so the rendered unit + readiness honestly reflect the 6.4.4 backend (Pitfall 2/3). |
