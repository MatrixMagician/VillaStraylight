# Phase 15 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

> **Scope (deliberately narrow).** This phase adds ONE read-only readout — cumulative
> prompt + generated token totals for the current configured model — into the EXISTING
> Performance panel of the control dashboard, alongside the existing live tok/s. Per
> CONTEXT.md **D-10** there is NO new API endpoint: the dashboard reads the same
> `status.Report` field that `villa status` exposes. This is NOT a new page, chart,
> time-series, navigation, or interactive control (all explicitly deferred — CONTEXT.md
> `<deferred>`). The contract below specifies how to render counts-only cumulative totals
> consistently with the established no-build, embedded, loopback-only SPA. **No new design
> tokens are introduced — every value below already ships in `dashboard.css`.**

## Design System

| Property | Value |
|----------|-------|
| Tool | none (no-build embedded SPA — plain HTML/CSS/vanilla JS served verbatim via `go:embed`; no toolchain, bundler, or framework — CLAUDE.md "no JS toolchain/bundler in the villa path") |

`dashboard.css` `:root` tokens, the `.metric-row` / `.metric-label` / `.metric-value`
classes, the `.muted` empty-state paragraph, and the `mutedP()` + `metricRow()` JS helpers.
It MUST NOT add new CSS custom properties, new color values, new font sizes, or any new
panel/section. The new totals render INSIDE the existing `#performance-body`.

## Color

Already declared in `dashboard.css:5-19` (60/30/10 split — reuse, do not add values):

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#0F1115` (`--bg-dominant`) | Page background, surfaces |
| Secondary (30%) | `#1A1D24` (`--bg-secondary`) | Panels (incl. Performance panel), header |
| Accent (10%) | `#3B82F6` (`--accent`) | Chat link, Switch button, focus rings ONLY |
| Destructive | `#EF4444` (`--destructive`) | Restart warning + connection-lost banner ONLY |

Accent reserved for: chat link, the model Switch primary button, the loaded-model
left-border, and keyboard focus rings — **as already established. This phase adds NO new
accent use.** The cumulative-total rows are plain neutral text: `--text-muted` for the
label, `--text-primary` (mono) for the value. The "no usage recorded yet" empty state uses
`--text-muted` via the `.muted` class (NOT a status color, NOT accent, NOT red).

Second semantic color: none required (this is a read-only display with no destructive
action and no failure state — an unavailable store is typed-Unknown muted text, not an
error).

**Honesty-by-construction color rule (D-05 / D-09):** When the usage field is absent
(`omitempty` → omitted from the `Report`), render the muted "no usage recorded yet"
copy in `--text-muted`. NEVER render a fabricated `0` total as a confident reading, and
NEVER use a green/ready status color to imply a real value exists. This mirrors the
existing Performance panel's `"Idle — no active generation."` / `"Unavailable"` discipline
(`dashboard.js:243-278`).

## Copywriting Contract

This phase has **no CTA and no destructive action** — it is a read-only counts display.
The contract below covers the labels, the unit, and the honest empty/unavailable states.

| Element | Copy |
|---------|------|
| Primary CTA | none (read-only display — no button, no interactive control) |
| Cumulative section sub-label | `cumulative usage` (a muted section caption preceding the two total rows, OR fold into the row labels below; planner/executor discretion within the existing `.metric-row` rhythm) |
| Prompt-tokens row label | `prompt tokens (total)` |
| Generated-tokens row label | `generated tokens (total)` |
| Value format | integer token count, thousands-grouped, rendered in `--font-mono` tabular-nums (e.g. `1,284,907`); append no unit suffix beyond the row label OR a trailing ` tokens` — keep it counts-only, never tok/s (those are the live-rate rows, which remain) |
| Per-model attribution | the totals shown are for the **current configured model** (the model the dashboard scrape keyed by `cfg.Model`); if the dashboard surfaces the model id it does so via existing identity copy — do NOT invent a new model-name treatment in this panel |
| Empty state heading | `No usage recorded yet` |
| Empty state body | `Cumulative token totals appear here once the dashboard has observed generation. Totals accumulate while the dashboard service is running.` |
| Unavailable state (scrape/store unreadable) | `Cumulative usage unavailable` (muted; same honesty as the existing `"Unavailable"` performance copy — never a fabricated 0) |
| Error state | not applicable — a missing/garbled store degrades to the Unavailable muted copy (typed-Unknown), never an error banner or red text |
| Destructive confirmation | none (no destructive action in this phase) |

**Copy constraints (privacy / zero-telemetry — D-11/D-12):**
- Render **counts only**. NEVER display any prompt or response text, message content, or
  per-request detail. The labels and values above are the entire surface.
- Do not imply any data leaves the box; do not add "sync", "upload", "report", or
  cloud-tinged verbs anywhere in this surface.

**Placement (D-10):** These rows render **inside `#performance-body`, alongside the
existing live tok/s rows** (live `generation` / `prompt` tok/s rows stay unchanged). The
cumulative totals are an ADDITIVE block in the same Performance panel — not a new panel,
not a new section heading, not new navigation.

## Interaction & State Contract (read-only)

The Performance panel renders from the `/api/metrics` poll for live rates and from the
`/api/status` poll for identity (D-01 precedent). The cumulative totals ride on the
`status.Report` field (D-09/D-10), so they are available on the SAME `/api/status` poll the
dashboard already runs every 2500ms — no new fetch, no new endpoint, no new cadence.

| State | Trigger | Render |
|-------|---------|--------|
| Available (totals present) | `Report.usage` present for the current model | Two `.metric-row` lines (prompt total, generated total), mono tabular values, alongside the live tok/s rows |
| No usage yet (field omitted) | `usage` absent via `omitempty` (store empty / dashboard just started) | Muted `No usage recorded yet` + body copy; NO fabricated 0 |
| Unavailable | status poll fails / field unreadable | Muted `Cumulative usage unavailable`; stale-while-revalidating dim applies via the existing global `.stale` rule on poll failure |

**No interactive elements.** No buttons, no inputs, no expand/collapse, no sorting, no
tooltips beyond what already exists. Updates are passive (re-rendered on the existing poll).

**Accessibility:** reuse the existing panel semantics (`#performance-panel` is already an
`aria-labelledby` section). The numeric rows are static text; `font-variant-numeric:
tabular-nums` prevents reflow on update. No new ARIA live-region is required (the panel
content updates in place at the existing poll cadence, matching the live tok/s rows).
