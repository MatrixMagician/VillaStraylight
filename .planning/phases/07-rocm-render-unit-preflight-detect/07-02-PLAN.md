<objective>
Deliver a reusable, refuse-with-remediation ROCm preflight verdict (PRE-06), driven by an externalized `go:embed`-updatable policy, that Phase 8's switch verb gates on and that is demoable now via `villa preflight --backend rocm` — off-hardware.

Three motions, ordered:
1. Externalize the v1.0 version floors into a `go:embed`'d `rocm-policy.json` (D-04/D-05). The migration MUST be a behavior no-op — existing preflight checks/goldens stay byte-identical; that is its acceptance test.
2. Add `RunROCm`/`RunROCmWithPolicy` ROCm checks reusing the existing `CheckResult`/`Tier`/`Status`/`pass`/`warn`/`fail` vocabulary (D-01) — NO new verdict type. Only a positively-detected bad state FAILs; any unevaluable signal degrades to WARN (D-02/D-15), honoring "biased not to over-block." Off-hardware almost every signal is Unknown→WARN.
3. Wire `villa preflight --backend rocm` to call `RunROCm` (D-03); the standalone `Run` path stays WARN-only and unchanged.

Purpose: the host-fitness gate the switch verb consumes, testable off-hardware now.
Output: an embedded policy, the migrated floors loader, the ROCm check pipeline, the CLI flag, and table tests covering both branches (known-bad→FAIL, unknown→WARN) per signal.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| build-time `go:embed` rocm-policy.json → binary | The policy is COMPILED INTO the binary, not loaded from a writable runtime path — not externally tamperable at runtime. A malformed embed is a build-time error. |
| `rocminfo`/host facts → RunROCm verdict | Tool output is already bounded (`maxToolOutput` 8 KiB) and parsed with fixed-arg exec in gpu_amd.go (reused, not re-rolled). |
