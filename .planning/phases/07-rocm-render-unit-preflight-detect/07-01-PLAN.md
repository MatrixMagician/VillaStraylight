<objective>
Render the ROCm `villa-llama.container` Quadlet unit as a pure delta over the Vulkan unit (ROCM-03 / D-09), frozen by a NEW byte-golden, with the existing Vulkan golden byte-for-byte unchanged.

The literal D-09 wording ("`AddDevice` becomes `[]string`") is INCOMPLETE. `backend_rocm.go.ContainerArgs` emits THREE render-affecting deltas over Vulkan, of which the parser currently handles only ONE:
- a SECOND `--device` (`/dev/kfd` then `/dev/dri`) — today last-wins, drops the first;
- a SECOND `--group-add` (`keep-groups` then `render`) — today last-wins, drops one;
- TWO `--env` flags (`HSA_OVERRIDE_GFX_VERSION=11.5.1` then `ROCBLAS_USE_HIPBLASLT=1`) — today there is NO `--env` case, so they are SILENTLY DROPPED.
Implementing only the device change yields a non-functional ROCm unit (no render group, no HSA/hipBLASLt env) that a naive golden still "freezes." This plan handles all three.

Purpose: produce the two off-hardware inputs Phase 8's switch verb gates on — the ROCm render output and its freeze — while proving additivity (Vulkan unchanged).
Output: a multi-value renderer, a `{{range}}`-based template, a new ROCm golden, and tests proving both the ROCm delta and Vulkan-byte-identity.
</objective>
